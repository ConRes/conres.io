## Context

PDFs generated by the `2026/generator` do not open in older Adobe Acrobat versions (confirmed: Acrobat CS6.5 on macOS 10.15). PDFs from the `2025/` implementation (deployed via `conres.io-staging`) open without issue in the same environment.

The key architectural difference: the `2026/generator` converts color in-browser using Color Engine and assembles pages by embedding source asset pages as Form XObjects into new blank pages. The `2025/` implementation directs users to convert color in Acrobat and works in-place on the source document.

### Source Test Material

- **2026/generator outputs** — docket and full test form PDFs with eciCMYK v2 and Canon iPR C10000VP profiles
- **2025/ legacy outputs** — finalized PDFs from both 2025-05-02 and 2026-03-28 runs (same code, different dates)
- **Source asset PDFs** — F9e, F9f, F10a from Illustrator

All test files stored in `temp/Generator Tests/`.

## Findings

### Structural Differences (2026 vs 2025 Output)

| Feature                    | 2025 (works)                                       | 2026 (fails)                                   |
| -------------------------- | -------------------------------------------------- | ---------------------------------------------- |
| Page content model         | Direct content (images, color ops)                 | Form XObjects (embedded pages via `Do`)        |
| Page boxes                 | MediaBox + TrimBox + BleedBox + CropBox            | MediaBox only                                  |
| Page ColorSpace resources  | `CS0: /DeviceCMYK`                                 | None (colors inside Form XObjects)             |
| Page ExtGState             | Full entries (OP, OPM, BM, CA, etc.)               | Empty dicts                                    |
| DestOutputProfile stream   | `/Alternate /DeviceCMYK /Filter /FlateDecode /N 4` | `/Length` only                                 |
| OCProperties               | None                                               | Content + Calibration OCGs                     |
| Document ID                | Present (from source Illustrator PDF)              | Missing                                        |
| XMP Metadata               | Present (from source Illustrator PDF)              | Missing (docket); present but stale (full PDF) |
| Residual ICCBased profiles | None (decalibrated)                                | sRGB, sGray from source Illustrator content    |

### PDF/X-4 Preflight Errors (Docket — 2026/generator)

| Rule   | Error                                | Fixable?                 |
| ------ | ------------------------------------ | ------------------------ |
| RUL118 | Page does not have TrimBox or ArtBox | Yes — set from MediaBox  |
| RUL123 | Document ID missing                  | Yes — generate and set   |
| RUL42  | PDF/X-4 entry missing or incorrect   | Needs XMP metadata work  |
| RUL54  | Metadata missing (XMP)               | Needs XMP metadata work  |
| RUL202 | Font not embedded (128 hits)         | Not fixable with pdf-lib |

### Key Insight from Preflight

The ICC profile stream fix (`/N`, `/Alternate`, `/Filter`) does **not** trigger any PDF/X-4 preflight error. The earlier RUL213/RUL208 findings were specific to PDF/X-4**p** (which expects a profile *reference*, not an embedded profile). The fix is good practice but is not a PDF/X-4 compliance issue.

### Franz's Results (Acrobat CS6.5, macOS 10.15)

| Variant                   | Opens | Renders     | Root Cause Isolation                            |
| ------------------------- | ----- | ----------- | ----------------------------------------------- |
| 00 Original               | Yes   | **Blank**   | Baseline — confirms the problem                 |
| 01 Traditional xref       | Yes   | **Blank**   | xref format is NOT the cause                    |
| 02 Fix Geometry           | Yes   | **Blank**   | TrimBox is NOT the cause                        |
| 03 Fix Profile            | Yes   | **Renders** | **ICC profile stream attributes ARE the cause** |
| 04 Fix Doc ID             | Yes   | **Blank**   | Document ID is NOT the cause                    |
| 05 All Fixes              | Yes   | **Renders** | Confirms profile fix is sufficient              |
| 06 All + Traditional xref | Yes   | **Renders** | Confirms profile fix is sufficient              |

### Root Cause: Confirmed

**The DestOutputProfile ICC stream missing `/N` and `/Alternate` causes older Acrobat to open the file but render blank pages.** No error message is shown — the pages simply don't render.

The fix: replace `context.stream(iccProfile, { Length: iccProfile.length })` with `context.flateStream(iccProfile, { N, Alternate })` in `setOutputIntentForPDFDocument`.

### Required Changes

1. **`services/PDFService.js`** — Fix `setOutputIntentForPDFDocument` to use `flateStream` with `/N` and `/Alternate` from the ICC header
2. **`2026/validator`** — Already has this as `fix-output-intent-profile` fix — can repair existing PDFs
3. **`../conres.io-staging`** — Deploy the PDFService fix so future generated PDFs are correct

### Previous Hypotheses (Resolved)

1. ~~Cross-reference stream format~~ — NOT the cause (variant 01 still blank)
2. **Bare ICC profile stream** — **CONFIRMED as root cause** (variant 03 renders)
3. ~~Missing Document ID~~ — NOT the cause (variant 04 still blank)
4. ~~Missing page boxes~~ — NOT the cause (variant 02 still blank)
5. ~~Combination~~ — Single fix (profile) is sufficient
