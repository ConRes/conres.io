**Created:** 2026-03-28
**Last Updated:** 2026-03-30

## Context

PDFs generated by the `2026/generator` fail to open in older Acrobat versions. Investigation revealed multiple structural issues (missing page boxes, bare ICC streams, orphaned OCG layers, missing metadata). A standalone validator/fixer tool is needed to:

1. Analyze PDF structure against a configurable rule set
2. Report pass/fail findings with human-readable explanations
3. Fix correctable issues upon user approval

The tool will eventually integrate into the `2026/generator` pipeline.

### Trigger

- Franz confirmed 2026/generator docket PDF does not open in Acrobat CS6.5
- Acrobat preflight (PDF/X-4) identified 5 error categories in 2026/generator output
- 2025/ legacy outputs pass in the same environment

## Design Decisions

### Architecture: Layered (Decided)

Three layers matching existing project patterns:

```
testing/iso/ptf/2026/
├── classes/
│   ├── configurations/
│   │   └── preflight-rules.json              # Declarative rule definitions
│   └── baseline/
│       └── pdf-preflight-validator.js        # Base check engine (no UI, no fix logic)
└── validator/
    ├── index.html                            # Entry point with drop zone
    ├── validator.js                          # Bootstrap module
    ├── bootstrap-worker-entrypoint.js        # Worker entry (same model as generator)
    ├── elements/
    │   └── pdf-validator-app-element.js      # Custom element (UI)
    └── classes/
        ├── pdf-preflight-fixer.js            # Fix operations (standalone)
        └── pdf-preflight-fixer-validator.js  # Extends validator, integrates fixer
```

### Class Hierarchy (Decided)

- **`PDFPreflightValidator`** (`classes/baseline/`) — Pure analysis engine. Takes PDFDocument + rules JSON, evaluates conditions by scope (document, page, object), produces structured report. No fix knowledge, no UI dependency. Reusable from experiments, CLI, and generator.
- **`PDFPreflightFixer`** (`validator/classes/`) — Fix operations only. Knows how to apply corrections (geometry, ICC attributes, document ID, OCG cleanup, metadata sync) to a PDFDocument.
- **`PDFPreflightFixerValidator`** (`validator/classes/`) — Extends validator, composes fixer. Can override severity based on fixer capabilities. Orchestrates two-phase flow: `validate()` → report → `fix(report, approvedFixes)`.

### Rules JSON Schema (Decided)

Follows `color-conversion-rules.json` pattern — declarative, with human-readable context:

- Rules grouped by category (page geometry, output intent, OCG, metadata, color spaces, fonts, etc.)
- Each rule has: `ruleId`, `pdfxReference`, `displayName`, `description`, `scope`, `severity`, `conditions`, `fixId`
- Conditions use Acrobat preflight property key namespace (`PAGE::HasTrimBox`, `DOC::HasDocumeID`, etc.)
- Fixes defined in a separate map — multiple rules can share a fix
- Severity is an object with context keys (extensible, specific keys deferred)

### UX Flow (Decided)

Two-phase:
1. **Analysis** — Drop PDF, run all checks, display read-only report
2. **Fix** — Show what will be changed, user approves, apply corrections

Worker thread for heavy operations (same model as generator, with same debugging controls).

### Scope (Decided)

All checkable categories — reporting pass/fail even where we can't fix. The full PDF/X-4p rule set (223 rules, 283 conditions) is the reference, with relaxations:

- If no output intent profile: offer to embed one, or skip output-intent-dependent checks
- Device/ICCBased color space constraints relaxed based on output intent profile

### Architecture: Async Generator with RegExp Scanning

Inspired by `~/.claude/meta/workspaces/scripts/utilities/packages/json-markup-parser` architecture:

- **Chunk-based streaming**: Read file in chunks, don't load entire PDF into memory for validation
- **RegExp forward scanning**: Match PDF structural markers (`obj`, `endobj`, `stream`, `endstream`, `<<`, `>>`, `%%EOF`, ICC `acsp` signature) on raw bytes
- **Async generator yields findings incrementally**: Each pass emits results as discovered
- **Multi-pass by concern**:
  1. **Structural scan** (lightweight, ~50 MB): xref, trailer, catalog, page tree, dicts only
  2. **Content stream validation** (selective, +1-5 MB): decode only content streams, validate operators
  3. **Image metadata validation** (dict-only, negligible): inspect image dicts without decoding pixels
  4. **Pixel data validation** (optional, one-at-a-time): decode each image, verify byte count, release after each
  5. **Ghostscript validation** (optional, separate): use pixel-stubbed version to validate structure through GS
- **Full pdf-lib load only when fixing**: The generator passes produce a report; pdf-lib is only loaded for the fix phase
- **Existing infrastructure reuse**: ColorSpaceUtils content stream parsing, ICCService header parsing, PDFService object traversal, GhostscriptService WASM

### Streaming Decompression Layer

`DecompressionStream('deflate-raw')` is a native browser API (baseline 2023). Its `readable` implements `Symbol.asyncIterator`, composing directly with async generators — zero glue:

```javascript
async function* inflateRaw(compressedBytes) {
    const ds = new DecompressionStream('deflate-raw');
    const writer = ds.writable.getWriter();
    writer.write(compressedBytes);
    writer.close();
    yield* ds.readable;
}
```

The full validation pipeline is one composed async generator chain: file chunks → object boundaries → selective decompression → operator/dict scanning → finding emission. Memory at any point is just the current chunk in flight.

Tested and confirmed (8/8 pass) in both Node.js 24.7.0 and Chromium 145 (via Playwright).
Firefox 115 (Franz's minimum) supports all required APIs: `DecompressionStream` (Firefox 113+), `ReadableStream[Symbol.asyncIterator]` (Firefox 110+).

Zero-buffer compression ratio: 1 MB zeros → 1,033 bytes (0.099%) — confirms pixel stub viability.

Separate effort to transition the generator from pako/zlib to Compression Streams API: see `2026-03-29-Compression-Streams-API-Adoption-Progress.md`.
