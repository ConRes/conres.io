# Embedding Missing Fonts — PROGRESS

**Created:** 2026-03-31
**Last Updated:** 2026-03-31
**Status:** Planning

---

## Context

PDF/X-4 requires all fonts to be embedded. PDFs generated by pdf-lib use `embedFont(StandardFonts.Helvetica)` which creates a reference to the standard font name without embedding the actual font data. Acrobat preflight reports this as RUL202 "Font not embedded (and text rendering mode not 3)."

The generator's docket pipeline resolves this by running the docket through `GhostscriptService.embedFontsInPDF()` — a full GS re-process pass. This works for the docket because we control it end-to-end.

The validator cannot use this approach. It handles arbitrary PDFs — including 16-bit images, complex color spaces, and structures that GS would alter or downsample. The validator must never run the actual PDF through GS WASM.

### Solution: GS as a Font Extraction Service

Use GS WASM to produce embedded font programs from a **proxy PostScript file**, then transplant the font streams into the original PDF without GS ever touching the original.

---

## Architecture

### Phase 1: Detection (Already Implemented)

The validator already detects unembedded fonts:

- Enumerates all Font dicts in the PDF
- Checks FontDescriptor for FontFile/FontFile2/FontFile3
- Reports each unembedded font with its name and location

### Phase 2: Proxy-Based Font Extraction

1. **Collect unembedded font names** from the PDF's font dictionaries
2. **Generate a minimal PostScript proxy** — one line per font, just enough to force GS to load and embed:

   ```postscript
   %!PS-Adobe-3.0
   /NimbusSans-Regular findfont 12 scalefont setfont 72 720 moveto (x) show
   /NimbusSans-Bold findfont 12 scalefont setfont 72 700 moveto (x) show
   showpage
   quit
   ```

3. **Run the proxy through GS WASM** with `-dEmbedAllFonts=true -dSubsetFonts=false`
4. **Parse GS stdout** for font loading messages to determine:
   - Which fonts GS found and loaded (e.g., `Loading NimbusSans-Regular font from %rom%Resource/Font/...`)
   - Which fonts GS could NOT find (substitution warnings or errors)
5. **Extract font program streams** (FontFile/FontFile2/FontFile3) and FontDescriptors from the proxy PDF using pdf-lib
6. **Report results**: which fonts were resolved, which are missing

### Phase 3: Transplant

1. **Register extracted font programs** as new indirect objects in the original PDF
2. **Update existing FontDescriptor entries** to reference the new font program streams
3. **Preserve everything else** — pages, images, content streams, metadata, 16-bit data untouched

### Phase 4: Missing Font Handling (Deferred)

When GS cannot find a font:

- **Report to user**: "Font X could not be embedded — not available in Ghostscript's font library"
- **Deferred — user-provided fonts**: Dialog showing missing fonts as individual file input fields, allowing users to drop/select font files from their system
- **Deferred — web font curation**: Curated list of free web fonts mapped to common PDF font names

## Constraints

- **The validator must NEVER run the original PDF through GS WASM** — GS alters PDF structure, downsamples 16-bit images to 8-bit, re-encodes streams, and can break complex content
- **GS is used only as a font extraction service** — produce font programs from a proxy, transplant the streams into the original PDF
- **Font program format**: GS produces CFF (Compact Font Format) in FontFile3. pdf-lib can register these as indirect streams. The FontDescriptor must be updated to point to the new FontFile3 reference
- **Subsetting**: `GhostscriptService.embedFontsInPDF()` accepts `{ subset }` option:
  - **Generator docket** (`subset: true`, default): GS subsets fonts to only the glyphs used in the docket. Safe because the docket contains all its own text.
  - **Validator proxy** (`subset: false`): GS produces full (non-subsetted) font programs. Required because the proxy PDF only renders sample characters — the original PDF may use any glyph. Transplanting a subset would leave characters missing.

## Proven Techniques

From the investigation work in this conversation:

| Technique                                       | Verified                                    | Notes                                                         |
| ----------------------------------------------- | ------------------------------------------- | ------------------------------------------------------------- |
| GS WASM FS access for fonts                     | No — `%rom%` is internal, not Emscripten FS | Fonts accessible only via PostScript `findfont`               |
| PostScript proxy → GS → PDF with embedded fonts | Yes (GS 9.56) / No (GS 10.05.1)            | PS `findfont` does not embed in GS 10; use pdf-lib proxy PDF instead |
| pdf-lib proxy PDF → GS `-dPDFX` → embedded fonts | Yes (GS 9.56 + 10.05.1)                   | `-dPDFX` forces embedding; `-dSubsetFonts=false` for full programs  |
| Font program extraction from GS-output PDF      | Yes                                         | CFF format in FontFile3, 77KB Regular, 80KB Bold              |
| GS `-dEmbedAllFonts=true -dSubsetFonts=false`   | Yes                                         | Full font programs, no subsetting                             |
| GS stdout font loading messages                 | Yes                                         | `Loading NimbusSans-Regular font from %rom%Resource/Font/...` |
| pdf-lib FontDescriptor inspection               | Yes                                         | Can read/write FontFile/FontFile2/FontFile3 refs              |
