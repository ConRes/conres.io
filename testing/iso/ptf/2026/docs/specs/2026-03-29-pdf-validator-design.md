# PDF Validator Tool — Design Spec

**Created:** 2026-03-29
**Status:** Draft
**Progress Tracker:** `testing/iso/ptf/2026/2026-03-28-VALIDATE-PDF-PROGRESS.md`

---

## 1. Purpose

A standalone browser-based tool that analyzes PDF structure against PDF/X-4 rules, reports findings with human-readable explanations, and fixes correctable issues upon user approval. It serves two immediate needs:

1. **Fix existing PDFs** generated by the 2026/generator that fail in older Acrobat versions
2. **Prevent future issues** by validating generator output before distribution

The tool will eventually integrate into the 2026/generator pipeline.

## 2. Architecture

### 2.1 Layered Structure

```
testing/iso/ptf/2026/
├── classes/
│   ├── configurations/
│   │   └── preflight-rules.json              # Declarative rule definitions
│   └── baseline/
│       └── pdf-preflight-validator.js        # Base check engine (no UI, no fix logic)
└── validator/
    ├── index.html                            # Entry point with drop zone
    ├── validator.js                          # Bootstrap module
    ├── bootstrap-worker-entrypoint.js        # Worker entry (same model as generator)
    ├── elements/
    │   └── pdf-validator-app-element.js      # Custom element (UI)
    └── classes/
        ├── pdf-preflight-fixer.js            # Fix operations (standalone)
        └── pdf-preflight-fixer-validator.js  # Extends validator, integrates fixer
```

### 2.2 Class Hierarchy

**`PDFPreflightValidator`** (`classes/baseline/pdf-preflight-validator.js`)
- Pure analysis engine
- Takes a `PDFDocument` (loaded by pdf-lib) and rules configuration
- Evaluates conditions by scope: document, page, object
- Produces a structured report (findings array + summary)
- No fix knowledge, no UI dependency
- Reusable from experiments, CLI tools, and the generator

**`PDFPreflightFixer`** (`validator/classes/pdf-preflight-fixer.js`)
- Fix operations only
- Methods for each fix category: geometry, ICC attributes, document ID, OCG cleanup, XMP metadata
- Takes a `PDFDocument`, applies corrections, returns what was changed
- No validation logic — just knows how to fix things

**`PDFPreflightFixerValidator`** (`validator/classes/pdf-preflight-fixer-validator.js`)
- Extends `PDFPreflightValidator`
- Composes a `PDFPreflightFixer` instance
- Overrides severity based on fixer capabilities (e.g., a fixable error can be presented differently)
- Orchestrates the two-phase flow: `validate()` → report → `fix(report, approvedFixes)` → fixed PDF

### 2.3 Why Three Classes

- The validator in `classes/baseline/` can be imported by the generator, experiments, or CLI without pulling in fix logic
- The fixer is independent — it can be used to batch-fix PDFs without running validation
- The fixer-validator is the integration point for the UI, combining both capabilities

## 3. Rules Schema

### 3.1 File Location

`classes/configurations/preflight-rules.json`

Follows the pattern established by `color-conversion-rules.json` in the same directory.

### 3.2 Schema Structure

```json
{
  "schemaVersion": "1.0",
  "profile": "pdf-x-4",
  "categories": [
    {
      "categoryId": "page-geometry",
      "displayName": "Page Geometry",
      "description": "Page box definitions required for prepress workflows.",
      "rules": [
        {
          "ruleId": "page-missing-trimbox-or-artbox",
          "pdfxReference": "RUL122",
          "displayName": "Page does not have TrimBox or ArtBox",
          "description": "Either ArtBox or TrimBox should be defined for pages used in prepress. PDF/X ISO standards require the presence of one of these boxes.",
          "scope": "page",
          "severity": {
            "default": "error"
          },
          "conditions": [
            { "property": "PAGE::HasTrimBox", "expected": true },
            { "property": "PAGE::HasArtBox", "expected": true }
          ],
          "logic": "or",
          "fixId": "set-geometry-from-mediabox"
        }
      ]
    }
  ],
  "fixes": {
    "set-geometry-from-mediabox": {
      "displayName": "Set page boxes from MediaBox",
      "description": "Sets TrimBox, BleedBox, and CropBox to the page's MediaBox values (full page, no bleed/trim/crop distinction).",
      "strategy": "copy-from-mediabox",
      "targets": ["TrimBox", "BleedBox", "CropBox"]
    }
  }
}
```

### 3.3 Key Schema Decisions

- **`fixId`** references a separate `fixes` map — multiple rules can share a fix, rules without `fixId` are check-only
- **`conditions`** use Acrobat's `property_key` namespace (e.g., `PAGE::HasTrimBox`) — direct mapping to preflight report data
- **`logic`**: `"or"` = any condition passing satisfies; `"and"` = all must pass; `"none"` = none should match (prohibition)
- **`scope`**: `"document"` | `"page"` | `"object"` — determines iteration target
- **`severity`** is an object with context keys — extensible, specific contexts defined per-rule as needed (not a global `"relaxed"` / `"strict"` toggle)

## 4. Validator Engine

### 4.1 Condition Evaluation

The validator maintains a registry of property evaluators:

```javascript
propertyEvaluators.set('PAGE::HasTrimBox', (pageNode, context) =>
    pageNode.lookup(PDFName.of('TrimBox')) !== undefined
);
```

Each evaluator is a function that takes the target object (page node, document catalog, indirect object) and returns a boolean.

### 4.2 Finding Structure

Each rule evaluation produces:

```javascript
{
  ruleId: 'page-missing-trimbox-or-artbox',
  status: 'fail',           // 'pass' | 'fail' | 'skipped'
  severity: 'error',        // resolved from context
  scope: 'page',
  location: { page: 3 },    // or { ref: '380 0 R' } for objects
  fixId: 'set-geometry-from-mediabox',
  details: {                 // rule-specific, for UI display
    hasTrimBox: false,
    hasArtBox: false,
    mediaBox: [0, 0, 792, 595.276]
  }
}
```

### 4.3 Report Structure

```javascript
{
  documentInfo: { pageCount, producer, pdfVersion, fileSize, ... },
  findings: [ ...all findings ],
  summary: {
    errors: 12,
    warnings: 3,
    passed: 45,
    skipped: 8
  }
}
```

### 4.4 Skippable Rules

Rules with dependencies that can't be evaluated get `status: 'skipped'`. Example: color space conformance checks that depend on the output intent — if no output intent exists, these rules are skipped rather than false-failing. The `skipped` count is surfaced in the summary.

## 5. Fixer Engine

### 5.1 Fix Categories (MVP)

| Fix ID | What It Does | Applies To |
| --- | --- | --- |
| `set-geometry-from-mediabox` | Sets TrimBox, BleedBox, CropBox to MediaBox values | Pages |
| `fix-output-intent-profile` | Adds `/N`, `/Alternate`, `/Filter` to DestOutputProfile stream, applies FlateDecode | Output Intent |
| `add-document-id` | Generates random 16-byte hex ID pair | Trailer |
| `strip-orphaned-ocg` | Removes OCG references not listed in OCProperties | Document-wide |
| `add-occd-name` | Adds Name entry to OCCD if missing | OCProperties |

### 5.2 Fix Approval Flow

1. Validator produces report with findings
2. UI displays report — fixable findings show a fix description
3. User selects which fixes to apply (or "Fix All")
4. Fixer receives the document + list of approved fix IDs
5. Fixer applies corrections, returns a changelog
6. UI offers the fixed PDF for download
7. User can re-validate the fixed PDF to confirm

### 5.3 Memory Management for Fix Phase

For PDFs over ~100 MB with image streams:
- Image pixel data is replaced with deflated zeros of the exact decoded pixel count before save
- This retains structural validity (same stream count, valid dicts) while dropping memory from ~2x file size to ~0.1x
- The report explicitly flags stubbed streams as "pixel data not validated"
- Streams that fail to decode during stubbing are flagged as errors, not silently skipped

## 6. Streaming Architecture (Phase 2)

Phase 2 replaces the pdf-lib-first approach with an async generator pipeline for bounded-memory validation:

**Pass 1 — Structural scan** (dict-only, ~file size memory):
- Parse xref/trailer, catalog, page tree
- Check geometry, document ID, output intent, OCProperties, XMP
- Covers ~60% of rules

**Pass 2 — Content stream validation** (selective decompress via `DecompressionStream('deflate-raw')`):
- Decode only content streams (<1 MB typically)
- Validate operators, q/Q nesting, resource references

**Pass 3 — Image/stream metadata** (dict-only, negligible extra memory):
- Image stream dicts, ICC profile headers (128 bytes)

**Pass 4 — Pixel validation** (optional, one-at-a-time):
- User opt-in, decode each image, verify byte count

**Pass 5 — Ghostscript** (optional, separate):
- Pixel-stubbed version for GS structure validation

Each pass yields findings incrementally. The UI displays results as they arrive.

The streaming pipeline uses `DecompressionStream('deflate-raw')` with `yield*` on `ReadableStream` — verified working in Node.js 24, Chromium 145, and compatible with Firefox 115 (Franz's minimum). See `2026-03-29-COMPRESSION-STREAMS-API-ADOPTION-PROGRESS.md`.

## 7. UI

### 7.1 Layout

Modeled on the generator UI (`2026/generator/index.html`):

- Drop zone for PDF files
- Report display with findings grouped by category
- Severity icons: error, warning, info, pass, skipped
- Fixable findings show the fix description and a checkbox
- "Fix Selected" / "Fix All" button
- Progress bar during analysis and fix phases
- Download button for fixed PDF

### 7.2 Execution Model

Same as generator:
- Worker thread for normal operation (non-blocking UI)
- Main thread path for debugging
- Same debugging controls (details disclosure, processing strategy selector)

### 7.3 Worker Protocol

**Incoming (UI → Worker):**
- `{ type: 'validate', taskId, pdfBuffer }` — start analysis
- `{ type: 'fix', taskId, pdfBuffer, approvedFixes }` — apply fixes

**Outgoing (Worker → UI):**
- `{ type: 'finding', taskId, finding }` — single finding (progressive display)
- `{ type: 'report', taskId, report }` — complete analysis report
- `{ type: 'fixed', taskId, pdfBuffer, changelog }` — fixed PDF ready
- `{ type: 'progress', taskId, stage, percent, message }` — progress update
- `{ type: 'error', taskId, message, stack }` — error

## 8. MVP Scope (Phase 1)

The MVP validates and fixes the specific issues found in 2026/generator output:

### 8.1 Rules Included

| Rule | Category | Capability |
| --- | --- | --- |
| Page missing TrimBox/ArtBox | Page Geometry | check-fix |
| Document ID missing | Document Structure | check-fix |
| XObject missing Subtype | Form XObject | check (crashes Acrobat) |
| OCG not in OCProperties | Optional Content | check-fix |
| OCCD missing Name | Optional Content | check-fix |
| DestOutputProfile bare stream | Output Intent | check-fix |
| Metadata missing (XMP) | XMP Metadata | check-fix |
| Document damaged | Document Structure | detect-throw |

The MVP `preflight-rules.json` contains ONLY these rules. Phase 2 expands it to all 223.

### 8.2 What the MVP Does NOT Include

- Full 223-rule coverage (Phase 2)
- Streaming async generator pipeline (Phase 2)
- Content stream operator validation (Phase 2)
- Font embedding checks (Phase 2)
- Color space conformance checks (Phase 2)
- XMP metadata sync fixes (Phase 2 — MVP only checks presence)
- Ghostscript validation (Phase 2)

### 8.3 Integration with Franz's Results

When Franz reports back with Compatibility 1A results:
1. Identify which variant(s) opened successfully → root cause confirmed
2. Apply that fix to `2026/generator` directly (e.g., `setOutputIntentForPDFDocument`, page assembly)
3. Add the fix to the MVP fixer so existing PDFs can be corrected
4. Deploy both to `../conres.io-staging`
